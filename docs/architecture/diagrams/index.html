monet Architecture

Multi-Agent Orchestration SDK

monet solves multi-agent coordination across machines you own — laptop, phone, VPS, Cloud Run. The key insight: orchestration and execution are fully decoupled by a queue. The server owns the DAG. Workers own the agent runtime. The queue is the only connection. The server is blind to everything except what you deliberately leak into graph state.

01

S1 — Local All-in-One

DEVELOPMENT

monet dev on your laptop. Aegra + Postgres + Redis run in Docker. pool="local" means workers run as background threads in the server process. No network, no remote. The simplest possible setup — same graph topology as production, all in one place.

Key Points

  • Single machine, Docker-backed Postgres + Redis
  • Workers as in-process background tasks (pool="local")
  • Same graphs as production: chat → planning → execution
  • MonetClient points to localhost:2026
  • Great for: quickstart, custom-graph examples, local dev loop

S2 — Self-Hosted Production

SELF-HOSTED

aegra serve on your infra + managed Postgres/Redis. Workers run as separate processes on machines you control. Queue is the only connection between server and workers — no agent runtime data ever crosses that boundary. Server sees DAG + HITL decisions only.

Key Points

  • Single tenant, one server + one or more worker machines
  • MONET_API_KEY shared by server and all workers
  • Pool name in monet.toml — workers register by pool
  • Workers own: model, tools, context injection, OTel export
  • Great for: CI/CD pipelines, dedicated infra teams

03

S3 — Split Fleet

MULTI-POOL

Same as S2 but N named worker pools — fast (Cloud Run), heavy (dedicated VPS), laptop (your own machine). Worker heartbeats populate a CapabilityIndex so the planner knows the fleet-wide capability set at plan time. Workers can be behind NAT — they poll outbound, no inbound holes.

Key Points

  • CapabilityIndex — planner sees all pools' capabilities
  • RoutingNode carries pool name — workers poll their queue slice
  • Planner composes across pools without knowing where agents run
  • No inbound firewall needed — polling works behind NAT on phone/laptop
  • Push pools (Lambda, Vercel) declared in config but dispatcher not yet built

04

S5 — SaaS Platform

VENDOR-HOSTED

The monet SDK (MIT) as a library imported by a downstream SaaS repo. The vendor builds user accounts, billing, and multi-tenant isolation. Vendor FastAPI app mounts monet.server.factory() and injects its own auth/tenant resolver — everything downstream is pure monet. Queue plane works today; control-plane primitives (per-tenant keys, workspace ID) are on the roadmap.

Key Points

  • Queue plane: works today — Redis pull model with tenant API key scoping
  • Auth swap boundary: vendor mounts monet factory, injects tenant resolver
  • Control plane primitives (Priority 1 roadmap): per-tenant MONET_API_KEY, tenant ID in path/state
  • Customers deploy their own workers — own model, tools, OTel export
  • monet stays MIT-licensed; commercial concerns isolated downstream

Core Design Concepts Across All Scenarios

Queue-Decoupled Orchestration

The Redis task queue is the sole coupling point between orchestration and execution. Workers poll for tasks — the server enqueues RoutingNode payloads (agent_id, command, depends_on). Agent code, model calls, tool results, and runtime data never cross the queue boundary. The queue uses a pull model — workers make outbound polls only, so NAT and firewalls are never an issue.

Orchestrator Blindness

The orchestration server (Aegra) is intentionally opaque to execution. It knows: the DAG structure, HITL decisions, artifact store pointers. It does NOT know: agent code, model choice, tool execution, LLM calls, runtime logs, hardware specs. The user controls exactly what leaks into ArtifactStore / graph state — and therefore what the orchestrator can see.

Capability-Based Pool Routing

Workers register with named pools via heartbeat. The CapabilityIndex accumulates the fleet-wide capability set so the planner can compose agent tasks across heterogeneous infrastructure — fast Cloud Run for quick research, heavy VPS for deep document synthesis, laptop for ad-hoc personal tasks. The planner writes pool names into RoutingNode — workers self-select their slice of the queue.

OpenTelemetry + User-Owned Observability

OTel is non-negotiable throughout the stack. The server emits traces for graph execution. Workers emit their own traces from their runtime environment and ship to their own exporter (Langfuse, Datadog, your OTLP endpoint). Trace continuity across the queue boundary is maintained via context propagation in the signal layer. The vendor in S5 never sees the customer's LLM call traces unless the customer opts in.
