monet — S5: SaaS Platform

VENDOR-HOSTED ORCHESTRATOR

monet SDK as MIT-licensed library — downstream SaaS repo builds multi-tenant product on top. Queue plane works today, control plane primitives in progress (roadmap).

Boundary: monet as MIT Library

  • • SaaS product (user accounts, billing, quotas, dashboards) lives in a downstream repo — NOT in monet
  • • Vendor ships a FastAPI app that mounts monet.server.factory() and injects its own auth/tenant resolver
  • • Everything downstream of that boundary is pure monet MIT code
  • • Keeps monet MIT-licensed and focused on orchestration correctness

Queue Plane: Works Today

  • • Redis/Upstash backend is fully pull-based — no inbound callbacks needed
  • • Tenant isolation via API key scoping in Redis
  • • Customer workers poll their vendor's queue just like S2/S3
  • • Works for customers behind NAT, on mobile, on laptops

Control Plane: Roadmap (Priority 1)

  • • Today: single global MONET_API_KEY — needs per-tenant key primitive
  • • Needs: tenant/workspace/project ID in request path, state, MonetClient.list_runs()
  • • See CLAUDE.md ## Roadmap Priority 1 for 5 required extension points
  • • Queue plane + auth swap boundary are already the right primitives
